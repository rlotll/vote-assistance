## 2. 배경 및 문제

20~30대 유권자는 정치 관심은 있으나 후보·공약 정보 접근 비용이 높아 참여율이 낮다. 짧고 직관적인 정보 소비 패턴에 맞춰 **선거구 단위로 공약을 비교하고, 모의투표로 의사결정을 연습하고, 실제 투표소까지 도달**하는 경로를 한 화면 단위로 압축 제공한다.

---

## 3. 목표 사용자

- **1차**: 20대 첫 투표자 — "내 선거구가 어디인지부터 모름"
- **2차**: 30대 정보 효율 추구형 유권자 — "공약을 짧게 비교해서 판단하고 싶음"
- **공통 특성**: 모바일 우선, 짧은 세션, 정치적 중립 정보 선호

---

## 4. 목표 / 비목표

### 4.1 Goals
- G1. 선거구 설정 → 투표소 확인 → 공약 비교 → 모의투표를 **한 앱 내에서** 완결
- G2. 후보·정당 정보를 **중립적**(기호 순)으로 제시
- G3. **사전투표 대상자**(주민등록 ≠ 거주지)를 명확히 분기 안내
- G4. 공식 데이터(국가선거정보 개방포털 API)만을 출처로 사용

### 4.2 Non-Goals
- 후보·정당 추천, 정치 성향 분석/매칭
- 모의투표 결과 저장·집계·외부 전송 (NF-03 기기 내 저장만)
- 회원가입·로그인·서버 측 개인정보 저장
- 비선거 기간 신규 데이터 제공 (API 제약)

## 6. 핵심 사용자 시나리오

1. **첫 진입**: S-01에서 D-day 확인 → "내 투표소" 진입 → S-02 선거구 설정 → S-03 투표소 위치 확인.
2. **의사결정**: S-04에서 후보 공약을 분야 필터로 비교 → 필요 시 S-05 정당 정책 비교 → S-06 모의투표로 선택 시뮬레이션.
3. **사전투표 대상**: S-02에서 거주지 다름 체크 → 사전투표 안내 박스/배지 노출 → 전국 사전투표소 안내.

흐름도 원본: `wireframe_doc.md` §5.

## 7. 기능 범위 (v1.0)

화면 단위로 묶고, 상세 요구사항은 ID로 추적한다.

| 화면 | 목적 | 포함 요구사항 | 우선순위 |
|---|---|---|---|
| S-01 메인 | 선거 D-day·빠른 메뉴 진입 | F-01, F-02 | 상 |
| S-02 선거구 설정 | 주소 기반 선거구 결정·사전투표 분기 | F-03~F-06 | 상 |
| S-03 투표소 안내 | 주변 투표소 지도/리스트 | F-07, F-08, EX-06 | 상 |
| S-04 공약 비교 — 후보자 | 분야 태그 필터 + 후보 카드 비교 | F-09~F-12, F-15 | 상 |
| S-05 공약 비교 — 정당 | 비례대표 정당 정책 비교(지방선거 한정) | F-09, F-13~F-15 | 상 (조건부) |
| S-06 모의투표 | 실제 투표용지 구성의 스텝형 시뮬레이션 | F-16~F-19 | 상 |

## 10. 제약 및 리스크

- **데이터 가용성**: 개방포털 API는 선거 기간에만 신규 데이터 제공 → 비선거 기간 UX 별도 설계 필요.
- **중립성**: 후보/정당 노출 순서·강조 색상은 코드 레벨에서 정렬 규칙 고정.
- **개인정보**: 주소·알림 설정은 `localStorage` 등 기기 내 저장만 허용.
- **선결 조건**: 선거구 미설정 상태에서 타 기능 접근 시 S-02로 유도.

---

## 11. 디자인 원칙

- **중립성**: 기호 번호 순서 외 정렬 금지.
- **접근성**: 최소 터치 영역·다크모드·스크린리더 대응.
- **사전투표 시각 구분**: 주황 계열(#FAEEDA/#633806) 배지로 일반 안내(초록)와 분리.
